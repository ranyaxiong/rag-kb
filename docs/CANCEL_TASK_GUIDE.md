# 任务取消功能使用指南

## 📋 功能概述

当上传大型文档（特别是扫描版PDF）时，后台处理可能需要较长时间。新增的任务取消功能允许用户在处理过程中随时停止任务，避免长时间等待。

## 🎯 使用场景

- **文档过大**：上传了超大文件，处理时间过长
- **扫描版PDF**：OCR识别耗时，需要中断处理
- **误操作**：上传了错误的文件，需要取消
- **资源优化**：释放服务器资源处理其他任务

## 🚀 使用方法

### 方法一：前端界面操作（推荐）

1. **上传文档后**，在上传成功提示中会显示两个按钮：
   - 📊 查看处理状态
   - 🛑 取消处理

2. **点击"🛑 取消处理"按钮**即可停止任务

3. **查看正在处理的文档列表**：
   - 在上传界面顶部会显示"🔄 正在处理的文档"展开框
   - 每个文档都有独立的"状态"和"取消"按钮

### 方法二：API调用

#### 1. 取消任务

```bash
POST /api/documents/cancel/{document_id}
```

**请求示例：**
```bash
curl -X POST "http://localhost:8000/api/documents/cancel/abc-123-def"
```

**响应示例（成功）：**
```json
{
  "success": true,
  "message": "任务已取消（未开始执行）",
  "document_id": "abc-123-def",
  "status": "cancelled"
}
```

**响应示例（任务正在执行）：**
```json
{
  "success": true,
  "message": "取消请求已发送，任务将在下一个检查点停止",
  "document_id": "abc-123-def",
  "status": "cancelling"
}
```

#### 2. 查询任务是否可取消

```bash
GET /api/documents/cancel-status/{document_id}
```

**响应示例：**
```json
{
  "success": true,
  "document_id": "abc-123-def",
  "status": "processing",
  "cancellable": true,
  "progress": 45,
  "message": "正在解析文档内容..."
}
```

## 🔧 技术实现

### 取消机制

任务取消采用**多检查点机制**，在处理流程的关键步骤检查取消标志：

1. **检查点 1**：开始处理前
2. **检查点 2**：文件移动前
3. **检查点 3**：文档解析前
4. **检查点 4**：文档解析后
5. **检查点 5**：向量化前
6. **检查点 6**：完成前

### 资源清理

取消任务时会自动清理：

- ✅ 临时上传文件
- ✅ 已移动的文档文件
- ✅ 已生成的向量数据（如果已存储）
- ✅ 任务状态记录

### 状态流转

```
queued → processing → cancelled
                   ↘ completed
                   ↘ failed
```

## ⚠️ 注意事项

### 1. 取消时机

- **任务未开始**：立即取消，状态变为 `cancelled`
- **任务执行中**：在下一个检查点停止，可能需要几秒钟
- **任务已完成**：无法取消

### 2. 数据一致性

- 取消后的文档**不会**出现在文档列表中
- 已生成的向量数据会被自动删除
- 文件会从服务器完全清除

### 3. 性能影响

- 取消操作本身非常快速（< 1秒）
- 不会影响其他正在处理的任务
- 释放的资源可立即用于新任务

## 📊 状态说明

| 状态 | 说明 | 可取消 |
|------|------|--------|
| `queued` | 任务已提交，等待执行 | ✅ 是 |
| `processing` | 任务正在执行 | ✅ 是 |
| `cancelling` | 取消请求已发送，等待停止 | ⏳ 进行中 |
| `cancelled` | 任务已取消 | ❌ 否 |
| `completed` | 任务已完成 | ❌ 否 |
| `failed` | 任务失败 | ❌ 否 |

## 💡 最佳实践

1. **大文件上传**：建议先测试小文件，确认系统正常后再上传大文件
2. **监控进度**：定期查看处理状态，及时发现问题
3. **合理取消**：如果文档确实需要处理，建议等待完成而非频繁取消
4. **资源管理**：同时处理的文档不宜过多，避免资源竞争
